# MiniCorteX

MiniCorteX is a small Python-based AI inspection pipeline inspired by a real manufacturing defect detection system. It accepts a product image, runs an AI-based visual inspection using Groq Vision, returns a structured JSON result, and generates a plain-English insight for a non-technical factory floor manager.

The project includes:

* Task 1: Core Python Inspection Service
* Task 2: LLM Conversational Insight Layer
* Task 3: Tech Lead Notes
* Task 4: FastAPI API, CSV logging, and Docker support

## Setup

Create and activate a virtual environment:

```bash
python3 -m venv venv
source venv/bin/activate
```

Install dependencies:

```bash
python -m pip install -r requirements.txt
```

Create a `.env` file in the project root:

```env
GROQ_API_KEY=your_groq_api_key_here
```

The API key is loaded using `python-dotenv` and is never hardcoded in the source code.

## Core Python Inspection Service

This feature is implemented in:

```text
src/inspector.py
```

For Feature 1, I implemented an LLM Vision-based inspection service using Groq. The service accepts an image file path, validates the file, converts the image into base64, and sends it to a vision-capable LLM with a structured inspection prompt.

The service returns a structured JSON result containing:

* image filename
* defect status
* confidence score
* defect type
* recommendation
* processing time in milliseconds

### Why I Chose LLM Vision Approach

I chose the LLM Vision approach because MiniCorteX is designed to provide AI-based product inspection and conversational defect analysis. Using a vision-capable LLM allows the system to detect different types of visible defects without training a custom model.

### Error Handling

The service handles:

* missing image files
* unsupported image formats
* API timeout or connection failure

### Unit Testing

The unit test mocks the LLM response instead of calling the real Groq API. This makes the test fast, stable, and independent of API cost or network availability.

## Feature 2: LLM Conversational Insight Layer

Task 2 is implemented in:

```text
src/llm_insight.py
```

Task 2 takes the structured JSON output from Task 1 and converts it into a plain-English explanation for a non-technical factory floor manager.

The module uses Groq's LLM API. The prompt is stored as a clear variable in `src/llm_insight.py`, so it is easy to review, edit, and maintain.

The insight layer explains:

* whether a defect was detected
* the defect type
* the confidence score
* a likely operational cause
* a practical recommendation

If the LLM API is unavailable, missing, or returns an error, the system returns a safe fallback message instead of crashing.

## Feature 3: FastAPI Endpoint, CSV Logging, and Docker

Task 4 is implemented in:

```text
src/api.py
```

For the bonus task, I added a FastAPI service that exposes the inspection pipeline through a REST API.

The API includes:

```text
GET /
POST /inspect
GET /latest-result
```

### POST /inspect

This endpoint accepts an uploaded image file, runs Task 1 inspection, generates the Task 2 factory manager insight, appends the result to a local CSV log, and returns everything in one JSON response.

Example response:

```json
{
  "inspection_result": {
    "image_filename": "product_001.jpg",
    "status": "defect_detected",
    "confidence_score": 0.87,
    "defect_type": "scratch",
    "recommendation": "Send the product for manual QA review.",
    "processing_time_ms": 1324.45
  },
  "factory_manager_insight": "A surface scratch was detected on the inspected part with 87 percent confidence. This may be related to tooling wear or rough material handling, so the part should be sent for manual QA review.",
  "logged": true,
  "log_file": "logs/inspection_logs.csv"
}
```

### GET /latest-result

This endpoint returns the latest inspection result generated by `POST /inspect` while the server is running.

### CSV Logging

After every successful inspection, the result is automatically appended to:

```text
logs/inspection_logs.csv
```

Each log entry contains:

* timestamp
* image filename
* defect status
* confidence score
* defect type
* shortened LLM insight summary

The CSV file is created automatically after the first successful inspection.

## Design Decisions

I kept the project structure simple and modular. `src/inspector.py` owns the core inspection logic for Feature 1, `src/llm_insight.py` owns the manager-friendly LLM explanation for Feature 3, and `src/api.py` exposes the workflow through FastAPI for Feature 4.

I used mocked LLM responses in the unit test so the test does not depend on network access, API cost, or external service availability.

For logging, I used a local CSV file because it is lightweight and easy to inspect during a timed challenge. In a production version, I would replace this with a database and stronger schema validation.
